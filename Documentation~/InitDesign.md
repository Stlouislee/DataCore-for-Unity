# DataCore — Unity Package 设计文档

**版本** 0.5.0 · 草稿  
**定位** Unity BI/数据分析应用的幕后数据基础设施

---

## 一、包的定位与边界

DataCore 负责"把数据从各种来源变成应用可以直接用的东西"。它不关心数据最终被怎么呈现，只负责数据从进入系统到准备好被消费这整个过程。

**DataCore 负责的事情：**

- 从各种外部来源读取数据，统一成内部标准格式
- 提供查询、筛选、连接、聚合、变换等完整计算能力
- 管理数据的缓存和刷新节奏
- 当数据发生变化时通知关心它的地方
- 为 AI Agent 提供独立的可读写、可持久化工作区
- 向 Agent 暴露数据的结构和统计摘要，支持推理

**DataCore 不负责的事情：**

- 图表和任何可视化渲染
- UI 控件和用户交互
- 业务逻辑（什么数据代表什么意义）
- AI Agent 本身的推理和自然语言理解

这条边界的意义是：应用层完全自由地决定怎么展示数据，AI Agent 完全自由地决定怎么分析数据，DataCore 只保证数据是干净的、是最新的、是可以高效操作的。

## 二、核心概念

### 数据源（Data Source）

数据的物理来源。可以是本地文件、远程 API、数据库，或是应用自己生成的模拟数据。每个数据源有唯一的名字，DataCore 通过这个名字来管理和查找它。

数据源分两类：

- **批量源**：每次请求返回一批完整数据，例如 CSV 文件、REST API。适合定期刷新的场景。
- **流式源**：数据持续推送进来，例如 WebSocket。DataCore 会持续监听并更新内部状态。

数据源始终是只读的。DataCore 从数据源读取数据，但不向数据源写回任何内容，除非应用层显式发起写回操作（见工作区一节）。

### 数据表（Data Table）

DataCore 内部统一的数据结构，类似关系数据库里的一张表：有固定的列定义（名称和类型），有若干行数据。无论数据从哪里来，进入 DataCore 后都统一变成数据表。

### 数据模式（Schema）

描述一张数据表的结构：有哪些列、每列是什么类型、是否允许为空。Schema 是 DataCore 与应用层之间的约定——应用可以依赖 Schema 来决定怎么处理数据，而不必关心数据从哪里来。

### 数据轮廓（Data Profile）

在 Schema 基础上更进一步，描述一张表的数据实际状况：数值列的最大值、最小值、均值；文本列的 distinct 值数量（值域小时直接列出所有可能值）；各列的非空率；整张表的行数和数据量估算。

数据轮廓主要供 AI Agent 使用。Agent 在构建查询之前需要先理解数据，轮廓是它推理的依据。

### 查询（Query）

对数据表执行的读取操作，由若干步骤组成，每个步骤都是可选的，步骤之间有固定的执行顺序。查询是无副作用的——执行查询不会改变任何数据的状态，每次查询得到的是一个独立的只读结果。

### 数据流（Data Stream）

区别于一次性查询，数据流是一种持续订阅机制。应用层订阅某个数据源或某个查询后，每当底层数据刷新，订阅方会自动收到新的结果，无需主动再查询一次。

### 变更事件（Change Event）

当数据源刷新后，DataCore 会描述"发生了什么变化"：是全量替换、还是有新增行、还是有行被修改。应用层可以利用这个信息做更精细的更新，而不必每次都全量重绘。

### Agent 工作区（Agent Workspace）

专门为 AI Agent 设计的独立可读写数据环境。工作区有自己的持久化存储，与所有数据源完全隔离，支持完整的增删改查操作，并记录所有操作的完整历史。

工作区是 DataCore 在只读数据层之上提供的一个独立层，两者并列存在，共享同一套查询与计算能力。

## 三、模块划分

### 3.1 数据源模块（Sources）

负责与外部世界打交道。每种数据来源对应一个适配器，适配器屏蔽了底层协议的差异，向内部统一交出一张数据表。

内置的适配器：

| 适配器 | 适用场景 |
|---|---|
| CSV / TSV | 本地静态文件，StreamingAssets 目录 |
| JSON | 本地文件或直接内嵌在项目里的结构化数据 |
| REST API | 通过 HTTP 定期拉取的远程数据 |
| WebSocket | 服务端主动推送的实时数据 |
| SQLite | 打包在应用里的本地数据库 |
| 模拟数据（Mock） | 开发和测试阶段使用，按规则生成随机数据 |

适配器是可扩展的。应用层可以实现自定义适配器接入任何数据来源，DataCore 不需要修改。

---

### 3.2 数据模型模块（Models）

定义 DataCore 内部流转的所有数据结构。这些结构是整个包的"语言"，所有模块之间通过这套结构传递信息。

**数据表（DataTable）**：行列结构，包含 Schema 定义和实际数据行。所有模块之间传递的数据都以数据表为载体。

**数据模式（Schema）**：列的名称、数据类型、是否可为空、原始来源（来自哪个数据源的哪个字段）。Schema 一旦确定，在该数据源的生命周期内保持稳定。

**数据轮廓（DataProfile）**：附加在 Schema 上的统计信息，见第 3.3 节。

**查询结果（QueryResult）**：查询执行后返回的只读快照，包含结果数据表、本次查询的元信息（执行耗时、扫描行数、是否命中缓存、原始行数），以及若查询失败时的详细错误信息。

**变更事件（DataChangeEvent）**：描述一次数据刷新的变化内容，包含变化类型（全量替换 / 追加行 / 修改行 / 删除行）和受影响的具体行数据。

**操作日志条目（WorkspaceLogEntry）**：工作区内每一步操作的记录，包含操作类型、操作对象、操作前后的数据快照、时间戳、以及发起该操作的来源标识。

---

### 3.3 数据轮廓模块（Data Profiling）

数据轮廓模块负责分析每张数据表的实际数据状况，生成供 Agent 和应用层参考的统计摘要。

#### 轮廓内容

对每一列，数据轮廓提供：

**数值列**：最小值、最大值、均值、中位数、标准差、零值数量、负值数量、非空率。

**文本列**：distinct 值的数量；当 distinct 值数量不超过一个阈值（默认 50）时，直接列出所有可能的值及其出现频次；最短和最长的文本长度；非空率。

**日期/时间列**：最早时间、最晚时间、时间跨度；非空率。

**所有列**：空值数量、空值率、总行数。

#### 轮廓的生成时机

轮廓不在每次查询时实时计算，而是在以下时机异步生成或更新：数据源首次加载完成后；数据源刷新且数据发生实质性变化后；应用层或 Agent 显式请求重新生成时。

轮廓的生成是后台任务，不阻塞正常查询。在轮廓尚未就绪时，DataCore 返回一个"轮廓未就绪"的状态，而不是阻塞等待。

#### 采样

对于大型数据表，全量计算轮廓成本较高。DataCore 支持基于随机采样的轮廓估算：从表中随机抽取一定比例的行，在样本上计算统计值，并在返回结果时标注这是基于采样的估算而非精确值。采样比例可配置，也可以指定采样的最大行数上限。

采样功能同时对外暴露给应用层和 Agent 使用——当 Agent 需要快速了解一张大表的数据面貌时，可以直接请求一个采样结果，而不必等待全量拉取。

---

### 3.4 查询与计算模块（Query & Computation）

这是 DataCore 计算能力的核心。所有对数据的读取和变换需求——无论简单还是复杂——都通过这个模块完成。计算全部在内存内进行，不依赖任何外部查询引擎。

工作区内的查询和只读数据源的查询走同一套执行引擎，能力完全对等。

#### 查询的执行顺序

一个查询由若干操作步骤组成，这些步骤有固定的执行顺序：

```
数据来源  →  JOIN  →  筛选  →  分组  →  聚合  →  计算列  →  窗口计算  →  行列转置  →  排序  →  截取
```

每个步骤都是可选的。最简单的查询可以只有"数据来源"，直接返回完整数据。

---

#### 数据来源

查询的起点。可以指向：

- 一张来自数据源的数据表
- 工作区内的任意一张表
- 另一个查询的结果（即子查询）
- 以上任意组合的连接结果

数据来源可以混合——一张工作区的派生表可以和一个数据源的实时表做 JOIN，两者在查询里地位对等。

---

#### 跨表连接（JOIN）

当数据分散在多张表里，需要按照某种关联关系合并在一起时，使用连接操作。

DataCore 支持四种连接类型：

**内连接（Inner Join）**：只保留在两张表里都能找到匹配行的数据。例如：订单表和用户表按用户 ID 连接，只返回有对应用户记录的订单。

**左连接（Left Join）**：保留左表的全部行，右表能匹配到的就填入，匹配不到的对应列留空。例如：列出所有订单，并尽量带上用户信息——即使某些订单的用户已被删除，订单本身也不丢失。

**右连接（Right Join）**：与左连接对称，保留右表的全部行。

**全外连接（Full Outer Join）**：两张表的所有行都保留，能匹配的合并，不能匹配的对应列留空。

连接条件通常是两张表的某一列相等（等值连接），也支持更复杂的条件表达式（如范围条件、多列组合条件）。

**多表连接**：支持多于两张表依次连接，后一个连接的输入是前一次连接的结果。例如：订单表 JOIN 用户表 JOIN 地区表。

**筛选的作用时机**：筛选可以在 JOIN 之后对合并结果生效，也可以在 JOIN 之前先对单张表做预筛选，减少连接时处理的数据量，提升性能。

---

#### 筛选（Filter）

从数据里只保留满足条件的行。

**单条件筛选**：对某一列的值进行判断，支持的判断类型包括：

- 等于 / 不等于
- 大于 / 大于等于 / 小于 / 小于等于
- 包含于某个值列表（IN）/ 不包含于（NOT IN）
- 范围判断（BETWEEN）
- 文本匹配：包含子串、开头是、结尾是、正则表达式
- 空值判断：是否为空（IS NULL / IS NOT NULL）

**复合条件筛选**：多个条件通过逻辑运算符组合：

- AND：所有条件都满足
- OR：至少一个条件满足
- NOT：条件取反

复合条件支持任意层级的嵌套，例如"(A AND B) OR (C AND NOT D)"这样的结构。

---

#### 分组（Group By）

把数据按照一列或多列的值分成若干组，后续的聚合操作在每个组内分别计算。

例如：按"地区"和"产品类别"两列分组，可以得到每个地区每个类别的独立汇总数据。

---

#### 聚合（Aggregation）

对每个分组（或整张表）计算汇总值。内置的聚合函数：

| 函数 | 含义 |
|---|---|
| 求和（Sum） | 组内某列所有值的总和 |
| 均值（Average） | 组内某列所有值的平均数 |
| 计数（Count） | 组内行数 |
| 去重计数（Count Distinct） | 组内某列去重后的值数量 |
| 最大值（Max） | 组内某列的最大值 |
| 最小值（Min） | 组内某列的最小值 |
| 首个值（First） | 组内按某列排序后的第一个值 |
| 末个值（Last） | 组内按某列排序后的最后一个值 |
| 标准差 | 组内某列值的标准差 |
| 方差 | 组内某列值的方差 |

聚合结果作为新的列附加到分组键上，构成聚合后的数据表。

**自定义聚合函数**：内置函数不够用时，应用层可以注册自定义聚合逻辑。DataCore 负责分组和调度，具体的计算规则由应用提供。

---

#### 计算列（Computed Columns）

在现有列的基础上派生出新的列，不改变原始数据。新列只存在于查询结果里，除非在工作区内显式将其持久化。

支持的计算类型：

**数值运算**：对一列或多列做加、减、乘、除、取模、幂次等运算。例如：用"销售额"列除以"数量"列，得到"单价"列。

**文本操作**：拼接、截取、大小写转换、去除空格、正则替换、格式化输出。

**条件表达式（Case When）**：根据条件给每行打上不同的标签或赋予不同的值。支持多个条件分支，最后一个分支可以作为默认值。例如：销售额大于 10000 标记为"大单"，大于 1000 标记为"中单"，其余为"小单"。

**类型转换**：把一列的值从一种类型转换成另一种类型，例如把文本格式的日期解析成日期类型，把数字格式化为带单位的文本。

**日期与时间运算**：提取年、月、日、星期、季度；计算两个日期之间的天数或月数差值；日期加减一段时间；格式化输出。

---

#### 窗口计算（Window Functions）

窗口计算是一种特殊的计算方式：它不像聚合那样把多行合并成一行，而是在保留每一行的同时，为每行附加一个基于"周围行"计算出来的值。

典型用途：

- **排名**：给每行打上排名编号（第 1 名、第 2 名……），可以在不同分组内分别排名，支持并列排名（相同值相同排名）和密集排名（并列后继续连续编号）两种方式
- **累计求和**：每行显示到当前行为止的累计总值，例如累计销售额
- **滑动均值**：每行显示前后若干行的平均值，即移动平均，常用于平滑时间序列数据
- **环比**：每行与上一行的差值或百分比变化
- **同比**：每行与去年同期行（按时间偏移定义）的差值或百分比变化
- **行偏移**：取当前行的前 N 行或后 N 行的某列值，常用于计算变化量

窗口的范围可以配置：可以是整个分组内的所有行、当前行之前的所有行（滚动窗口）、或者固定数量的前后若干行（滑动窗口）。

---

#### 行列转置（Pivot / Unpivot）

**行转列（Pivot）**：把某一列的不同值展开成多个列，另一列的值作为各新列的单元格内容。例如：把按月汇总的销售数据，从"月份"作为一列、每行一个月的格式，转成以每个月为一列的宽表，便于横向比较。

**列转行（Unpivot）**：Pivot 的逆操作，把多个列折叠成两列（一列存原列名、一列存值）。常用于把宽格式的数据转成适合聚合计算的长格式。

---

#### 排序（Order By）

对结果按一列或多列排序，每列可以独立指定升序或降序。排序在计算列和聚合之后执行，因此可以按计算列或聚合结果的值排序。空值的排列位置（放最前还是最后）可以单独指定。

---

#### 截取（Limit / Offset）

只取结果的一部分：

- **Limit**：只返回前 N 行
- **Offset**：从第 N 行开始返回（与 Limit 配合实现分页）

---

#### 子查询（Subquery）

把一个查询的结果作为另一个查询的数据来源。例如：先查出每个地区的销售总额，再在这个结果上筛选出总额超过某个阈值的地区，再把这些地区和产品维度表 JOIN 起来查明细。

子查询让复杂的分析逻辑可以分层表达，而不必把所有逻辑压缩在一个查询里。子查询可以任意层级嵌套。

---

#### 查询结果的错误反馈

当查询执行失败时，DataCore 返回的错误信息是结构化的、可机读的，而不只是一条文本消息。错误信息包含：

- 失败发生在查询的哪个步骤（筛选、JOIN、聚合……）
- 失败的原因分类：列名不存在、类型不兼容、JOIN 条件歧义、数据源不可达
- 当列名不存在时，附上该表实际存在的列名列表
- 当类型不兼容时，说明期望类型和实际类型分别是什么

这套结构化错误反馈的主要目的是让 AI Agent 能够自我纠错：Agent 收到错误后可以根据具体原因调整查询，而不是直接把错误抛给用户。

---

#### 查询优化

复杂查询（尤其是包含 JOIN 和多层筛选的查询）在执行前会经过一个优化步骤：

- 尽早执行筛选，减少后续操作处理的数据量
- 把 JOIN 前的预筛选自动下推到连接操作之前执行
- 对被频繁 JOIN 的列建立内部哈希索引，加快匹配速度
- 识别并复用相同子查询的结果，避免重复计算

优化对应用层和 Agent 完全透明——描述"想要什么结果"，DataCore 决定"怎么最高效地算出来"。

### 3.5 Agent 工作区模块（Agent Workspace）

> **实现状态（v0.5.0）**：工作区的核心功能已实现为 `IWorkspace` 接口和 `Workspace` 类，作为 `DataCoreStore.Workspace` 属性提供。当前覆盖：Register（IDataSet / 字典数据 / 自动命名）、Get（自动 fallback 到 store）、Describe / DescribeAll（元数据自省）、Summary、Rename / Clone / Remove / Clear 生命周期管理、来源追踪（Store / Derived / Imported）、内存保留策略（Strong / Weak / Auto）、懒缓存、异步 API。操作日志、检查点、回滚、混合查询等高级功能尚未实现，见下方完整设计。

工作区是 DataCore 为 AI Agent 提供的独立可读写数据环境。每个工作区有自己的持久化存储，与所有数据源完全隔离，支持完整的增删改查操作，并记录所有操作的完整历史。

#### 工作区的定位

数据源层是只读的、干净的，代表"外部世界的事实"。工作区是可读写的、可变的，代表"Agent 正在进行的工作"。这两层并行存在，互不干扰。

应用层可以读取工作区里的任何内容，工作区对应用层不是黑箱，但只有通过 DataCore 的工作区接口才能修改工作区内的数据。

#### 工作区内的表类型

**快照表（Snapshot Table）**：把数据源在某个时间点的查询结果"拍"下来，存入工作区，变成一张可以修改的本地副本。之后 Agent 对这张表的任何操作都不会影响原始数据源。快照记录了它来自哪个数据源、拍摄时间，以及生成快照时所用的查询条件。

**派生表（Derived Table）**：Agent 执行查询或计算后，把结果命名并持久化到工作区。派生表是真实存在的、可持久化的表，后续可以像普通数据表一样被引用、被修改、被进一步查询。派生表记录了它是由什么查询生成的。

**临时表（Scratch Table）**：完全由 Agent 从零创建的表，不来自任何数据源，也不是任何查询的结果。Agent 可以往里面写任意结构的数据，用于分析过程中产生的标注、分类结果、人工补充的信息等。

#### 工作区内的写操作

工作区内支持以下写操作，这些操作仅对工作区内的表生效，不影响任何数据源：

**行级操作**：插入新行、按条件修改行的字段值、按条件删除行。

**列级操作**：新增列（可以是空列，也可以是基于其他列计算得来的初始值）、重命名列、删除列、修改列的数据类型（进行类型转换）。

**表级操作**：新建表、重命名表、复制表（保留数据）、清空表（保留结构）、删除表。

**批量写入**：从查询结果或另一张表批量写入数据，支持覆盖写（替换整张表内容）和追加写（在现有行后追加）两种模式。

#### 工作区与数据源的混合查询

工作区内的表和数据源的表可以在同一个查询里混合使用。例如：把工作区里 Agent 标注过的客户分层表，和数据源里的实时订单数据做 JOIN，得到一个结合了 Agent 分析结果和最新数据的视图。

这种混合查询不需要任何额外配置，数据来源对查询引擎透明——工作区的表和数据源的表在查询里地位完全对等。

#### 工作区的生命周期

**创建**：Agent 在开始一项任务时创建工作区，并指定名称和描述。同一时间可以存在多个工作区，互相独立。

**工作**：Agent 在工作区内自由读写。工作区从创建起持续持久化，所有写操作实时落盘，不会因为会话结束或应用关闭而丢失。

**挂起与恢复**：对话结束或任务暂停时，工作区保持当前状态。下次 Agent 可以直接打开同名工作区，从中断处继续，工作区内的所有表和数据完好保留。

**查阅**：应用层随时可以读取工作区里的任何表，就像读取普通数据源一样。

**写回（可选）**：如果 Agent 在工作区里做的修改需要推送回某个原始数据源，这是一个独立的、需要显式发起的操作，不会自动发生，需要应用层确认授权后才能执行。

**归档与清除**：任务完成后，工作区可以被归档（保留数据但标记为非活跃）或清除（释放存储空间）。

#### 操作日志

工作区内每一个写操作都记录到操作日志里，日志条目包含：

- 操作类型（插入行 / 修改行 / 删除行 / 新建列 / 删除列 / 新建表 / 删除表……）
- 操作对象（哪张表、受影响的行数和列名）
- 操作前的数据快照（供回滚使用）
- 操作时间戳
- 来源标识（标明是 Agent 推理链路中的哪一步发起的，便于将操作与 Agent 的推理过程对应）

#### 可撤销性

任意一步操作都可以回滚，工作区恢复到该操作执行前的状态。支持回滚到任意历史节点，而不仅仅是最后一步。回滚本身也记录在操作日志里。

#### 检查点（Checkpoint）

Agent 可以在任意时刻对工作区的当前状态打一个命名检查点。之后可以选择直接回滚到某个检查点，而不必逐步撤销中间的每一步操作。

检查点适合用在两种场景：完成一个阶段性结果时做一次保存；准备尝试一个不确定的分析路径之前先打点，出了问题直接回来。

---

### 3.6 核心引擎模块（Core Engine）

DataCore 的协调中枢，负责把各模块连接起来。自身不包含业务计算逻辑，只负责调度和协调。

**注册与管理**：记录哪些数据源和工作区存在，管理它们的生命周期（初始化、刷新、挂起、销毁）。

**查询路由**：接收查询请求，判断数据来源（数据源 / 工作区 / 混合），交给查询与计算模块执行，把结果返回给调用方。

**缓存**：为每个数据源维护一份内存缓存，避免重复拉取。缓存策略可按数据源单独配置：批量源常用"有效期缓存"（TTL），大表常用"最近最少使用"（LRU）策略控制内存占用上限。工作区的数据不经过这层缓存，直接读写持久化存储。

**调度**：决定什么时候刷新数据源。支持三种模式：按固定时间间隔自动刷新、由外部事件触发刷新、以及完全手动控制。调度只作用于数据源，不影响工作区。

**事件广播**：数据更新后（无论是数据源刷新还是工作区写操作），通知所有订阅了相关数据的地方。广播是松耦合的——发送方不知道谁在监听，监听方不需要持有数据源或工作区的引用。

---

### 3.7 公开 API 模块（Public API）

应用层和 Agent 与 DataCore 交互的唯一入口。把所有内部复杂性封装起来，对外暴露稳定的操作接口。

**数据源操作**：注册 / 注销数据源；读取数据源的当前 Schema 和数据轮廓；执行一次性查询；订阅持续数据流；手动触发刷新。

**工作区操作**：创建 / 打开 / 关闭工作区；列出当前所有工作区及其状态；在工作区内执行读写操作；查阅操作日志；创建 / 恢复检查点；归档 / 清除工作区；发起写回到数据源的操作。

**跨源查询**：执行涉及数据源和工作区混合数据的查询。

公开 API 的稳定性是优先级最高的设计目标。内部实现可以迭代替换，但 API 的形态不应随意变动，避免破坏已经接入 DataCore 的应用。

---

### 3.8 编辑器工具模块（Editor）

仅在 Unity 编辑器中存在，不影响打包后的运行时体积。

**查询构建器**：在编辑器里以可视化方式构建和测试查询，实时预览查询结果，在开发阶段验证查询逻辑是否正确。

**Schema 与轮廓检查器**：在 Inspector 面板里查看某个数据源的 Schema、数据轮廓统计摘要以及当前缓存状态。

**工作区浏览器**：列出所有工作区，查看每个工作区内的表结构和数据内容，查阅完整操作日志，手动创建检查点或执行回滚。

**数据源配置器**：用图形化界面配置数据源的连接参数（URL、文件路径、刷新频率、缓存策略等），生成对应的配置资产（ScriptableObject）。

## 四、数据流向

### 常规读取路径

```
外部来源
  └─→  适配器（数据源模块）
         └─→  数据表（标准化）
                └─→  缓存层
                       └─→  查询引擎
                              └─→  查询结果 / 数据流
                                     └─→  应用层
```

### 数据刷新路径

```
调度器触发 / 外部事件 / 手动触发
  └─→  适配器重新拉取
         └─→  新数据表写入缓存，生成变更事件
                └─→  事件广播
                       └─→  所有订阅方收到通知，各自决定如何响应
```

### Agent 工作路径

```
Agent 接收用户指令
  └─→  请求数据轮廓（了解数据结构和分布）
         └─→  创建 / 打开工作区
                └─→  从数据源导入快照（可选）
                       └─→  在工作区内执行查询、写入、变换操作
                              └─→  每步操作实时持久化，同步写入操作日志
                                     └─→  结果交给应用层展示
                                            └─→  （可选）结果写回原始数据源（需授权）
```

### 混合查询路径

```
涉及多个数据来源的查询请求
  └─→  查询路由（核心引擎）
         ├─→  从缓存取数据源数据
         └─→  从持久化存储取工作区数据
                └─→  在查询引擎内合并执行（JOIN / 筛选 / 聚合……）
                       └─→  统一返回查询结果
```

## 五、配置与扩展

### 配置方式

DataCore 的所有运行时配置以 ScriptableObject 存储在项目里，可以纳入版本控制，也可以在不改代码的情况下调整参数。每个数据源有独立的配置资产，工作区的存储路径和容量上限也通过配置控制。

### 扩展点

| 扩展点 | 说明 | 典型场景 |
|---|---|---|
| 自定义数据源适配器 | 实现数据源适配器接口 | 接入内部私有 API、消息队列、专有数据库 |
| 自定义聚合函数 | 注册自定义聚合计算逻辑 | 业务特定的统计指标、加权平均等 |
| 自定义缓存策略 | 实现缓存策略接口 | 特殊的内存管理需求、自定义失效规则 |
| 自定义写回适配器 | 实现写回接口 | 把工作区修改推送回特定外部系统 |
| 工作区操作钩子 | 注册操作日志监听器 | 审计、同步、触发外部通知 |

## 六、设计原则

**幕后优先**：DataCore 不持有任何 Unity UI 对象的引用，不感知场景结构，不假设数据会被怎么展示，也不假设 Agent 的推理方式。

**源数据不可变**：数据源层永远只读，是"外部事实"的镜像。所有可变性都收敛到工作区，两者的边界清晰、不可逾越。

**推送而非轮询**：应用层通过订阅获得数据更新，DataCore 在数据就绪后主动推送。工作区的写操作同样触发事件广播，订阅方无需轮询。

**只读输出**：DataCore 交给应用层的所有数据结构都是只读快照，应用层不能直接修改。写操作必须通过工作区接口显式发起。

**增量感知**：DataCore 尽量描述"变化了什么"而不是只说"有变化"，无论是数据源刷新的变更事件，还是工作区写操作的日志条目，都携带足够的细节供调用方做精细响应。

**Agent 可观测性**：Agent 对数据的所有操作都是可追溯的。操作日志、检查点、数据来源记录，共同保证工作区的每一个状态都有据可查、可以回溯、可以解释给用户。

**结构化错误**：错误信息是可机读的结构体，不是纯文本字符串，确保 Agent 能够自动解析并据此调整行为。

**配置驱动**：刷新策略、缓存策略、工作区存储路径、轮廓采样比例等运行时参数都通过配置控制，不需要修改代码。

**内部封装**：除公开 API 外，所有内部实现对应用层和 Agent 不可见。调用方无法绕过公开 API 直接操作引擎内部状态或工作区存储。

## 七、未来扩展方向（待定）

以下方向在当前版本不做实现，但设计时预留了扩展空间：

**数据转换管线**：在数据从适配器进入缓存之前，允许挂载自定义的处理步骤（清洗、格式化、补全缺失值、字段映射）。与工作区的区别是这一层在数据进入 DataCore 时就执行，对所有下游透明。

**增量 JOIN**：当流式数据源持续更新时，只对受影响的行重新执行 JOIN，而非全量重算。适合实时数据量较大的场景。

**查询结果二次缓存**：对高频且完全相同的查询结果做缓存，避免重复计算。当前版本对数据源做缓存，这一项是对查询结果做缓存，粒度更细。

**持久化查询（Saved Query）**：把常用查询保存为有名字的可复用对象，应用层和 Agent 通过名字引用，不必每次重新描述完整的查询结构。

**工作区版本分支**：类似 Git 的分支概念，允许从一个工作区检查点创建分支，在分支上做实验性修改，之后选择合并回主线或丢弃。适合 Agent 需要同时探索多个分析路径的场景。

**跨工作区引用**：允许一个工作区的查询引用另一个工作区里的表，实现多 Agent 协作或多阶段分析结果的复用。
